# Chat Frontend Facts

**Maven module location**: The frontend Maven module descriptor lives at `java/chat-frontend/pom.xml`, but the npm project, build script, and generated `dist/` assets stay under `frontends/chat-frontend/`.

**LangChain event compatibility**: `useSseStream` and `chat-panel` normalize both canonical rich events (`eventType`) and raw LangChain envelopes (`event`, `data`) into the frontend `ChatEvent` model via `src/lib/langchain-events.ts`, so live streams and replayed `history/lc` entries render consistently.

**Required agent-app endpoint surface**: The UI currently uses these agent-app routes (directly or via generated client service wrappers): `POST /chat/{conversationId}`, `POST /v1/conversations/resume-check`, `GET /v1/conversations/{conversationId}/resume`, `POST /v1/conversations/{conversationId}/cancel`, conversation proxy routes (`GET/PATCH /v1/conversations/{id}`, `GET /v1/conversations` with `archived=exclude|include|only`, `GET /v1/conversations/{id}/entries`, `GET /v1/conversations/{id}/forks`, `POST /v1/conversations/search`), sharing routes (`GET/POST /v1/conversations/{id}/memberships`, `PATCH/DELETE /v1/conversations/{id}/memberships/{userId}`, `GET/POST/DELETE /v1/ownership-transfers`, `POST /v1/ownership-transfers/{transferId}/accept`), and attachment routes (`POST /v1/attachments` multipart only, `GET /v1/attachments/{id}/download-url`, `DELETE /v1/attachments/{id}`, `GET /v1/attachments/download/{token}/{filename}`), plus `GET /config.json`.
**Attachment rendering**: Stored history attachments should use `attachmentId` to call `GET /v1/attachments/{id}/download-url?disposition=inline|attachment`; never put authenticated attachment content endpoints directly in `<img>`/`<a>` elements because the frontend authenticates with bearer headers, not cookies.
**Attachment URL safety**: Attachment renderers must pass both external `href` values and server-returned download URLs through the local safe URL helper before assigning them to `window.open`, `<img src>`, or temporary download anchors. Only `http:` and `https:` URLs are allowed, and `image/svg+xml` is not treated as an inline image preview.
**Optional profile context surface**: `GET /v1/profile-context`, `GET /v1/profile-context/inputs`, and `PUT /v1/profile-context/inputs` are optional chat-app endpoints, not part of the baseline frontend contract. Feature-detect them with direct fetch calls; hide the user-menu Manage memory entry only when `GET /v1/profile-context` returns 404, and show it for successful empty profiles such as `{ "exists": false, "content": "" }`. Only show profile input editing inside that screen when the inputs endpoint succeeds.
**Conversation sidebar ordering**: Preserve the order returned by `GET /v1/conversations`; do not apply a client-side sort in `src/App.tsx`, or the sidebar can drift from the server’s pagination order.
**Conversation sidebar tree**: `src/App.tsx` loads top-level sidebar items from the roots conversation list, caches parent/child relationships for visited conversations, and uses `GET /v1/conversations/{conversationId}/children` for sidebar sub-navigation items in `src/components/chat-sidebar.tsx`.
**Archive UX**: Sidebar archive filtering is server-driven via `archived=exclude|include|only`, and archiving a conversation uses `PATCH /v1/conversations/{id}` with `{ archived: true }` rather than a delete route.
**Unsaved conversation gotcha**: New-chat IDs are generated client-side before the first message persists the conversation. Gate sidebar `getConversation`/`children` queries on the same known/resolved conversation check used by `chat-panel`, or the agent app can log 404s for brand-new conversations.
**Sidebar children invalidation**: `conversation.created` SSE payloads do not include `startedByConversationId`, only the new conversation ID and group ID. Invalidate the `["conversation-sidebar-children"]` query family broadly on conversation-created events so selected parents refresh their child sub-navigation.
**Tool call replay gotcha**: Replayed tool-call history can store args under `arguments`, and merged completed tool cards must preserve args from `BeforeToolExecution` while showing output from the correlated `ToolExecuted` event with the same ID.
**ContentFetched UI policy**: The chat UI normalizes but does not render `ContentFetched` events; retrieval context should stay hidden while assistant text, thinking, and tool events remain visible.
**Fork navigation and history paging**: the chat panel loads `/forks` as a lightweight `{conversationIds, forkPoints}` index, then loads the selected ancestry path with `/entries?forks=none&tail=true` and prepends older `beforeCursor` pages. Do not restore `forks=all` full-group entry loading or client-side ancestry reconstruction.
**Event-stream invalidation key**: Live entry invalidation must target the `["conversation-path-messages", conversationId, ...]` query family used by `chat-panel`, not a standalone `["entries", conversationId]` key or the visible message list will stay stale.
