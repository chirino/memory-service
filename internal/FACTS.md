# Internal Module Facts

**Conversation channel naming**: The canonical agent-state conversation channel is `context` (OpenAPI string / gRPC enum `CONTEXT`); `/v1/memories` remains the separate episodic memory API surface.
**Conversation context API boundary**: Per-conversation `context` sync/list behavior lives under `/v1/conversations/:conversationId/entries` and `/entries/sync` plus the corresponding store methods in `internal/plugin/route/entries` and `MemoryStore`; `internal/plugin/route/memories` is only for the separate episodic `/v1/memories` APIs.
**Conversation ID type rule**: Conversation IDs are arbitrary non-empty strings across REST, gRPC, stores, caches, and vector payloads. Example apps may still generate UUID-shaped conversation IDs, but code must not parse/validate conversation IDs as UUIDs. Conversation group IDs remain `uuid.UUID`.

**Go cache plugin architecture**: The Go service selects cache backends through `internal/registry/cache` and currently injects a `MemoryEntriesCache` into store loaders before datastore initialization. `cfg.CacheType` is shared by both the memory-entries cache plugins (`internal/plugin/cache/*`) and the response-recording locator-store selection in `internal/resumer/locator_store.go`, so new cache kinds usually need both an entries-cache implementation and locator-store behavior.

**GORM `record not found` log-noise rule**: If a `record not found` log line is found, treat expected-miss lookups as noise and refactor those call sites from `First(...).Error` to `Limit(1).Find(...)` with `RowsAffected` checks. Keep `First` for true not-found error paths; don't use global logger suppression.

**Error observability**: All 500-level errors MUST produce a full stack trace in the server logs. Never swallow exceptions silently — always log the stack trace for server errors.

**Embedded OPA Rego syntax**: Rego policies using `allow if { ... }` must import `future.keywords.if`; missing this import can fail policy loading and accidentally disable route-level authorization if startup is not fail-closed.

**Default episodic OPA policy**: Built-in authz is deny-by-default and only allows `["user", <subject>, ...]` namespaces. Built-in attribute extraction persists plaintext `{"namespace":"user","sub":<subject>}` in `policy_attributes`, and built-in search filter injection enforces both keys for non-admin callers.

**Episodic indexer pending rows**: `registry/episodic.PendingMemory.Namespace` is a single RS-delimited string (`\x1e`), not `[]string`; tests/mocks must provide encoded namespace values.

**Postgres append auto-create race**: Concurrent `AppendEntries` calls for the same missing conversation ID can race on root conversation/group creation and hit `23505` on `conversation_groups_pkey`. The append path must treat that unique-violation as a concurrent-create win, reload the conversation, and continue instead of returning 500.
**Postgres root-conversation upsert rule**: In `createConversationWithID`, root/start-from conversations must insert `conversation_groups` and `conversations` with `ON CONFLICT DO NOTHING` semantics. Catching `23505` after `Create(...)` is too late in Postgres because the surrounding transaction is already aborted and later reloads fail with `25P02`.
**Postgres duplicate-key diagnostics**: When conversation/group auto-create paths hit `23505`, the store now emits structured warning logs with `constraint`, `table`, and Postgres `detail` plus `userID`, `conversationID`, and `conversationGroupID` so CI flakes can be mapped back to scenario-specific IDs.
**oapi route registration gotcha**: Do not call generated `RegisterHandlers(...)` while other manual registrations for the same endpoints are mounted; Gin panics on duplicate `METHOD + PATH` registrations.
**Serve wrapper routing architecture**: `internal/cmd/serve/server.go` now exposes Agent/Admin HTTP endpoints through generated wrappers wired in `internal/cmd/serve/wrapper_routes.go`; runtime legacy route mounting in `serve` has been removed.
**Memories wrapper-native migration**: `/v1/memories*` endpoints are bound through generated wrappers backed by `internal/plugin/route/memories/wrapper_adapter.go`, which directly delegates to memories route logic instead of proxying through the legacy router.
**Transfers wrapper-native migration**: `/v1/ownership-transfers*` endpoints are bound through generated wrappers and handled directly via exported transfer route helpers (`internal/plugin/route/transfers/transfers.go`) instead of proxying through the legacy router.
**Memberships wrapper-native migration**: memberships endpoints under `/v1/conversations/:conversationId/memberships` are bound through generated wrappers and handled directly via exported membership route helpers (`internal/plugin/route/memberships/memberships.go`) instead of proxying through the legacy router.
**Attachments wrapper-native migration**: `/v1/attachments*` endpoints are bound through generated wrappers and handled directly via exported attachment route helpers (`internal/plugin/route/attachments/attachments.go`) instead of proxying through the legacy router.
**Attachment disposition behavior**: User and admin direct download, download-url, and signed token endpoints accept `disposition=inline|attachment`; when omitted, routes do not set `Content-Disposition` so browsers choose from content type. S3 signed URLs must include disposition via `registry/attach.SignedURLOptions` at presign time, not by appending response header overrides after signing.
**History attachment reference normalization**: `internal/plugin/route/entries.resolveAttachmentRefs` preserves `attachmentId` in stored history content and does not populate `href`; for linked cross-entry/fork references it rewrites `attachmentId` to the newly created attachment record ID, then links that record to the new entry.
**Signed attachment download auth parity**: `/v1/attachments/download/:token/:filename` must remain unauthenticated (token-auth only). In wrapper mode, register this endpoint with a wrapper that has no auth middleware; otherwise BDD signed-download scenarios fail with 401.
**Encrypted attachment size sentinel**: Attachment backends treat `maxSize < 0` as "caller already enforced the logical plaintext limit". The encrypted attachment wrapper uses this to stream ciphertext without imposing an encrypted-size ceiling.
**MSEH v2 stream nonce layout**: AES-CTR attachment streams use a 16-byte nonce whose leading bytes encode a stable key ID derived from the DEK. This lets `dek`, `vault`, and `kms` choose the correct rotated key before streaming decryption.
**Search wrapper-native migration**: `/v1/conversations/search`, `/v1/conversations/index`, and `/v1/conversations/unindexed` are bound through generated wrappers and handled directly via exported search route helpers (`internal/plugin/route/search/search.go`).
**Wrapper auth middleware gotcha**: Wrapper-native endpoints must run auth (and for memories, client-id) as wrapper middlewares in `internal/cmd/serve/wrapper_routes.go`; otherwise role checks can fail even when tokens are valid.
**Unix-socket listener validation**: Listener selection conflicts must use explicit flag/env detection (`cmd.IsSet(...)`), not non-zero resolved port values, otherwise the default port `8080` falsely conflicts with `--unix-socket`. When a Unix-socket parent directory is missing, the serve layer creates it as `0700`; if it already exists and is group/world accessible, startup must fail fast instead of chmod-ing it.

**TLS self-signed certificate opt-in**: Omitting `MEMORY_SERVICE_TLS_CERT_FILE` and `MEMORY_SERVICE_TLS_KEY_FILE` only generates an ephemeral self-signed certificate when `MEMORY_SERVICE_TLS_SELF_SIGNED=true` / `--tls-self-signed` is set. Dev launch surfaces that enable TLS without cert files must set this explicitly.
**OIDC self-signed issuer opt-in**: `MEMORY_SERVICE_OIDC_TLS_INSECURE_SKIP_VERIFY=true` / `--oidc-tls-insecure-skip-verify` only affects outbound OIDC discovery and JWKS fetches in `internal/security/auth.go`; it does not change listener TLS or general HTTP client behavior.

**gRPC recorder disconnect cleanup**: In `ResponseRecorderServer.Record`, if the stream fails with gRPC/ctx `CANCELED` or `DEADLINE_EXCEEDED` after a recorder has been created, call `recorder.Complete()` before returning so the locator/cache registry entry for that conversation is removed.

**gRPC cancel semantics**: `ResponseRecorderServer.Record` subscribes to `resumer.CancelStream(conversationID)` once the first chunk sets `conversation_id`; when cancel is requested it completes the recorder (removing locator/cache registry) and returns unary `RecordResponse{status=RECORD_STATUS_CANCELLED}`.
**Fork ancestry nil stop-point semantics**: In entry ancestry filtering (Postgres, SQLite, and Mongo), a nil `forkedAtEntryId` means "exclude all inherited entries from that ancestor and jump directly to the child conversation". This makes the first newly appended message the first visible entry in the fork.
**BDD Mongo SQL-skip semantics**: `internal/bdd/steps_sql.go` treats `nil` rows as "skip SQL assertions"; `MongoTestDB.ExecSQL` intentionally returns `nil, nil`, so SQL verification steps in shared feature files do not validate datastore state on Mongo runs unless parallel Mongo-specific assertions are added.
**Resettable DB schema source**: Fresh PostgreSQL and SQLite test/dev databases are created from `internal/plugin/store/postgres/db/schema.sql` and `internal/plugin/store/sqlite/db/schema.sql`. When the contract intentionally breaks and DBs will be reset, update those schema files directly instead of adding compatibility migrations.
**BDD Mongo query variable expansion**: Mongo query docstrings can contain Mongo `$` operators/field paths (for example `$ne`, `$match`, `$task_body.path`). Use brace-only expansion (`${var}`) for scenario variables; do not run generic `os.Expand` over whole Mongo query strings or `$ne`/`$foo` will be treated as missing variables.
**Godog report format switch**: BDD report writers honor `GODOG_REPORT_FORMAT`; default is `junit` (`.xml`), and setting `GODOG_REPORT_FORMAT=cucumber` writes Cucumber JSON (`.json`) under `GODOG_REPORT_DIR`.
**Cucumber report chunking tool**: `go run ./internal/cmd/create-cucumber-reports` scans `*/reports/*.json`, normalizes array/object report roots, and writes chunked combined files to `.artifacts/cucumber-report-###.json` (default max 50 features per file).
**BDD backend harness pattern**: Each datastore/backend gets its own `internal/bdd/cucumber_<backend>*.go` runner plus a matching `TestDB` implementation; shared feature files live under `internal/bdd/testdata/features*`, and backend-specific coverage is added with extra feature subdirectories instead of forking the whole suite.
**BDD gRPC event runner**: There is no standalone `TestFeaturesGrpc` runner. Common gRPC event scenarios under `internal/bdd/testdata/features-grpc/` run through `TestFeatures`, and the Postgres outbox runner includes `sse-events-grpc.feature` for outbox-enabled gRPC coverage.
**`internal/bdd` isolation strategy**: general user-scoped runners still use `bddScenarioConcurrency()`. All SQLite BDD runners provision a dedicated memory-service server plus SQLite file per scenario, while the common serial PostgreSQL and Mongo runners keep one shared server/datastore instance and run serial-only common features with `Concurrency = 1`.
**BDD scenario identity isolation**: `internal/testutil/cucumber` gives each `internal/bdd` scenario a short suffix for both canonical test users and agent `clientId`s. The harness rewrites known user IDs through request paths/bodies, gRPC prototext, and JSON/text responses, and it suffixes `X-Client-ID` values in `I am authenticated as agent with API key ...` so shared caches and client-scoped context entries do not collide across concurrent scenarios.
**BDD API-key wording gotcha**: The common non-OIDC BDD step `I am authenticated as agent with API key ...` uses `auth_testfixtures` + `ModeTesting` `X-Client-ID` identity, not production `X-API-Key` validation against `MEMORY_SERVICE_API_KEYS_<CLIENT_ID>`. Add focused production-mode auth tests when real API-key header behavior matters.
**BDD table parsing gotcha**: `godog.Table.Rows` includes the header row used in feature files. Step parsers that expect data rows must explicitly skip headers such as `| role | scope |`.
**BDD role-user config**: non-OIDC BDD runners configure static role users with wildcard suffixes like `alice-*` / `dave-*`, and the auth layer plus gRPC search-role checks honor `*` suffix prefix matches. That lets isolated subjects such as `alice-1a2b3c4d` keep admin/auditor/indexer roles inside concurrent scenarios.
**BDD ordinary-user isolation tests**: In `internal/bdd`, `alice` is admin, `charlie` is auditor, and `dave` is indexer (including isolated `*-<suffix>` subjects). Use users such as `bob` and `erin` when a scenario must prove ordinary user-to-user isolation.
**Mongo GridFS upload race**: `internal/plugin/attach/mongostore` serializes `Store(...)` calls with a mutex because the shared `go.mongodb.org/mongo-driver/v2` `GridFSBucket` first-write/index-bootstrap path races under concurrent uploads and can corrupt BDD attachment content under `-race`.
**BDD OIDC user provisioning**: `internal/testutil/testkeycloak` provisions scenario-local OIDC users on demand through the Keycloak admin API, and `deploy/keycloak/memory-service-realm.json` grants the seeded `alice` test user `realm-management` `realm-admin` so those provisioning calls can run over the imported test realm.
**S3 testcontainer image pinning**: `internal/testutil/tests3` should use an immutable `localstack/localstack@sha256:...` image reference, not `:latest`; Localstack `latest` can change under CI and fail during startup before the S3 BDD runner even creates the bucket.
**CGO SQLite runtime**: Runtime build paths now use `CGO_ENABLED=1` with `sqlite_fts5 sqlite_json` in both the repo `Dockerfile` and `deploy/dev/air.toml`. Any new SQLite-specific runtime code should assume CGO is available in dev and container builds.
**SQLite container base image**: The production `Dockerfile` must stay on a glibc-based image (currently UBI 9), not Alpine/musl. `sqlite-vec` needs SQLite development headers and fails to compile cleanly against musl in the multi-arch container build used by CI.
**Runtime version source**: Docker builds set `main.Version` with `-ldflags "-X main.Version=<version>"`; release builds pass the release version, and main-branch CI builds pass `github.sha` for the `latest` image. `main` passes that value into `internal/runtimeversion`, which is used by capabilities, the `version` subcommand, and the turn-traces processor. Do not reintroduce `MEMORY_SERVICE_VERSION` as a separate runtime-version source.
**REST tx-scope helper**: Route handlers that need `MemoryStore` or `EpisodicStore` transaction intent should use `internal/plugin/route/routetx` so the transaction-scoped context is attached to Gin’s request context before downstream store calls.
**Direct episodic fetches are write-scoped**: `GET /v1/memories` must use `EpisodicStore.InWriteTx`, not read scope, because the handler increments usage counters in the same workflow.
**Episodic archive parity**: Memory APIs now follow the same archive model as conversations. `GET /v1/memories`, `POST /v1/memories/search`, and `GET /v1/memories/namespaces` accept `archived=exclude|include|only`, archived memories remain directly readable, and the current memory state is the latest row per `(namespace,key)` rather than only rows with `archived_at IS NULL`.
**Episodic search filter injection gotcha**: Current `PolicyEngine.InjectFilter` merges caller and OPA `attribute_filter` values into a plain map, so same-key constraints are overwritten. Enhanced search work needs a normalized/conjoined filter representation before adding richer operators.
**Episodic event semantics**: Memory archive operations are represented as `update` lifecycle events at `archived_at`; `ListMemoryEvents` no longer exposes a `delete` kind for archive PATCHes.
**Episodic semantic search archive behavior**: Semantic `/v1/memories/search` honors `archived=exclude|include|only` too. Archived memory vectors are retained for manual archive (`deleted_reason=1`), vector backends filter by archive state before ranking, and hydrated results are post-filtered against the memory rows for correctness.
**Background store workers are tx-scoped**: `BackgroundIndexer`, `TaskProcessor`, `EvictionService`, `AttachmentCleanupService`, `EpisodicIndexer`, and `EpisodicTTLService` now declare store read/write intent via `InReadTx` / `InWriteTx` instead of calling store methods on an unscoped context.
**SQLite conversation vector backend**: `internal/plugin/vector/sqlitevec` uses a regular `entry_embeddings` table plus `sqlite-vec` scalar distance functions, not a separate `vec0` virtual table. That keeps schema/migrations simple and works with the shared GORM SQLite handle.
**SQLite episodic vector backend**: `internal/plugin/store/sqlite/episodic_store.go` stores episodic `memory_vectors.embedding` as serialized float32 BLOBs and queries them with `vec_distance_cosine(...)` plus SQLite JSON1 filters, mirroring the conversation-side `sqlite-vec` scalar-function approach.
**SQLite shared handle policy**: `internal/plugin/store/sqlite/shared.go` augments every SQLite DSN with `_foreign_keys=1`, `_journal_mode=WAL`, `_busy_timeout=5000`, and `_synchronous=NORMAL`, then uses a small multi-connection pool (`8` open / `4` idle). The shared handle is still process-wide; this avoids deadlocking request-scoped store transactions against vector-store local transactions on a single physical connection.
**SQLite startup fallback**: `internal/plugin/store/sqlite` now creates parent directories for file-backed DB paths before opening SQLite and probes both FTS5 and `sqlite-vec` support on the shared handle. When either extension is unavailable, migrations skip the corresponding optional schema and search/index paths degrade to empty/no-op behavior instead of failing startup.
**Infinispan vector-store split**: `internal/plugin/vector/infinispan` is a separate REST API v3 integration, not the RESP-backed cache/event-bus path. It needs Infinispan `16.1+`, defaults to `MEMORY_SERVICE_VECTOR_INFINISPAN_URL=http://localhost:11222`, and only falls back to the shared `MEMORY_SERVICE_INFINISPAN_URL/USERNAME/PASSWORD` values for connection reuse.
**gRPC tx-scope parity**: `internal/grpc/server.go` now declares `InReadTx` / `InWriteTx` around `MemoryStore` and `EpisodicStore` calls so the SQLite datastore can be registered globally without unscoped gRPC access paths.
**Episodic memory CAS**: gRPC and REST `PutMemory` / `UpdateMemory` support `expected_revision`; stores persist a monotonic per-`(namespace,key)` `revision` on each memory version and return `ErrMemoryRevisionConflict` as gRPC `ABORTED` or REST `409`.
**gRPC actor policy**: User-facing gRPC episodic memory requests may carry `RequestActor.on_behalf_of_user_id`; server code uses it only to derive the effective `PolicyContext.UserID`. OPA input does not include a separate on-behalf-of field or an `act_as` operation.
**Admin episodic memory writes**: Admin REST/gRPC memory write/update APIs are authorized by admin role/scope/justification and bypass episodic OPA authz. Admin puts still run OPA attribute extraction with a neutral admin `PolicyContext` (`UserID` empty, admin role).
**Admin episodic memory REST guard**: New `/admin/v1/memories` write handlers must call `ensureAdmin` in the handler, not rely only on generated proxy permission/scope checks; non-OIDC fixture users can otherwise reach admin memory writes when no OIDC scope gate is configured.
**Admin memory BDD runner**: `admin-memories-rest.feature` and `admin-memories-grpc.feature` use raw bearer admin/user fixture auth and therefore run under `TestFeaturesAuthFixturesSQLite` with the `auth_testfixtures` build tag, not under the broad default runners.
**Event-stream payload parity**: Membership event notifications use `data.user` across REST SSE and gRPC stream consumers. Conversation `deleted` notifications come from hard-delete paths (for example eviction), not archive PATCHes, and must carry a pre-captured `members` list because membership lookups are empty after the delete transaction commits.
**Event-stream normalization for 090**: REST SSE and gRPC event contracts now reserve `after` / `detail` replay parameters and use normalized business action names (`created`, `updated`, `deleted`) instead of legacy action-specific verbs such as `appended`, `added`, or `removed`. Entry and response event payloads must include `data.conversation_group`.
**Postgres outbox relay**: PostgreSQL outbox replay now uses opaque numeric `event_seq` cursors assigned by the `pgoutput` logical-decoding relay. `AppendOutboxEvents(...)` only persists rows (`event_seq` starts `NULL`), request-path `eventstream.PublishEvents(...)` is suppressed when Postgres outbox replay is enabled, and a single advisory-lock leader batches `event_seq` materialization per committed transaction before republishing the durable business event.
**Postgres outbox relay startup logging**: Initial relay bootstrap errors are returned to `serve.StartServer(...)` and should not also emit `Postgres outbox relay stopped` warnings; keep WARN logs for post-start background failures only.
**Postgres outbox relay shutdown noise**: `internal/plugin/store/postgres/outbox_relay.go` treats shutdown-time reconnect errors (`context canceled`, `connection refused`, `unexpected EOF`, closed network connection) as normal exit conditions. Keep those suppressed so testcontainer teardown does not emit misleading relay warnings.
**Postgres test harness replication**: `internal/testutil/testpg.StartPostgres` starts the PG18 test container with `wal_level=logical`, `max_replication_slots`, and `max_wal_senders` set so Postgres outbox relay tests can create a logical replication slot/publication without extra bootstrap.
**Postgres BDD task timestamps**: `PostgresTestDB` task helpers use Go-side UTC timestamps for `created_at` / `retry_at` and ready-task comparisons. Do not mix database `NOW()` with Go-side assertions, because testcontainer clock skew can make retry scheduling checks flaky.

**Response event detail mode**: `detail=full` enriches `conversation` and `entry` events only. `response` events intentionally keep the same compact payload as `detail=summary` on REST SSE, admin SSE, and gRPC streams.
**gRPC full event payload shape**: `EventStreamService.SubscribeEvents` always carries JSON bytes in `EventNotification.data`; `detail=full` swaps in marshaled internal `model.Entry` / `ConversationDetail` JSON for entry/conversation events, not generated proto `Entry` or `Conversation` messages. Full-detail enrichment preserves the original summary `conversation_group` as `conversationGroupId` for processor correlation, while internal struct fields like `ClientID` that remain `json:"-"` are still omitted.
**gRPC admin checkpoint auth**: `AdminCheckpointService` requires the admin role. If an authenticated client ID is present, it must match the checkpoint `client_id`; mismatches are returned as `NOT_FOUND`.
**Processor lifecycle API**: Event processors should expose an in-process `Start...(...)->Running...` API with `Wait` and `Shutdown` methods, then have CLI commands call that API. BDD tests should use the same API with injectable event/checkpoint clients and span sinks instead of shelling out.
**Turn-trace BDD wiring**: The Postgres outbox BDD runner includes `features-pg-outbox/turn-traces-processor.feature`; its steps start the processor through `turntraces.StartProcessor` against the scenario `grpcAddr` and capture spans with an injected sink, while checkpointing still goes through the real gRPC `AdminCheckpointService`.
**Turn-trace local dev**: `task dev:turn-traces-processor` starts the Docker Compose `memory-service` plus its backing services and Langfuse, stops the compose-managed `turn-traces-processor`, and runs `./memory-service process turn-traces` through `deploy/dev/air.toml` with Air CLI overrides against the compose-published `localhost:8082`.
**Turn-trace Langfuse naming**: The processor's Langfuse trace/root span name is configurable with `--langfuse-name` / `MEMORY_SERVICE_TURN_TRACES_LANGFUSE_NAME` and defaults to `memory-service.turn`.
**Turn-trace child span shape**: When a turn includes context entries, the processor emits a child `memory-service.llm` generation span under the root turn span. Its Langfuse input is the latest-epoch context messages up to the closing AI history entry, excluding a trailing assistant message; its output is that trailing assistant message when present, otherwise the closing AI history entry text.
**Turn-trace event filters**: `EventStreamService.SubscribeEvents` now defaults entry events to the `history` channel, so the turn-traces processor must explicitly subscribe with `entry_channels=["history","context"]` or it will miss context entries and skip `memory-service.llm` child generation spans.
**Entry list upper bounds**: REST and gRPC entry listing support `upToEntryId` / `up_to_entry_id` to return entries at or before a caller-visible bound entry. Use it with `channel=context` and an epoch filter to reconstruct agent context as of a specific history response entry while avoiding later async context writes.
**Turn-trace history content types**: Treat `history` and `history/<subtype>` entries as history. `history/lc4j` AI entries may omit top-level `text`; extract output from `Completed.aiMessage.text` or concatenate `PartialResponse.chunk` events.
**Docker Compose Langfuse wiring**: The root `compose.yaml` runs Langfuse against the existing Postgres container using a separate `langfuse` database from `deploy/keycloak/postgres-init-keycloak.sql`; it also reuses the existing Redis and MinIO services, adds ClickHouse, and runs `turn-traces-processor` from the same Memory Service image with `entrypoint: ["/memory-service"]`.
**Stream phase markers**: Event streams now emit `{"event":"phase","kind":"stream","data":{"phase":"replay"}}` before replay and `{"event":"phase","kind":"stream","data":{"phase":"live"}}` when tailing live events, so consumers can tell when they have reached the live tail.
**Event replay scope rule**: Outbox replay reads and replay-capability probes must run inside `MemoryStore.InReadTx`; SQLite outbox queries panic when `ListOutboxEvents` is called without a tx scope.
**Admin stats split**: `/v1/admin/stats/summary` is datastore-backed and does not need Prometheus, while the other `/v1/admin/stats/*` endpoints remain Prometheus-backed time-series queries.
**Episodic memory admin access split**: `/admin/v1/memories`, `/admin/v1/memories/{id}`, `/admin/v1/memories/search`, and `/admin/v1/memory-namespaces` are admin-or-auditor read APIs that can return decrypted memory values across users. Operational memory admin routes are admin-only and live under `/admin/v1/memory-policies`, `/admin/v1/memory-index`, and `/admin/v1/memory-usage`; force delete remains `DELETE /admin/v1/memories/{id}`.
**Auth identity boundary**: Normal non-admin user/agent APIs require a resolved user identity for ownership, membership, and policy filtering. Admin/operational APIs may authorize a client-only identity when configured client roles grant admin/auditor/indexer access; this can coexist with OIDC user auth when service principals use `X-API-Key`. Checkpoint APIs further restrict authenticated client IDs to matching checkpoint keys.
**Admin audit prefixes**: `internal/security/logging.go` audits and enforces admin justification for both `/v1/admin/...` and `/admin/v1/...`. Keep REST justification enforcement centralized in `AdminAuditMiddleware`; gRPC admin services must enforce justification separately.
**Admin SSE replay wiring**: `internal/cmd/serve/wrapper_routes.go` now passes `MemoryStore` into `internal/plugin/route/admin/events.go`, so `/v1/admin/events` supports the same outbox-backed `after` replay and `detail=full` enrichment shape as the agent SSE route.
**Admin justification policy owner**: `internal/security/logging.go` `AdminAuditMiddleware` is the single REST enforcement point for `MEMORY_SERVICE_ADMIN_REQUIRE_JUSTIFICATION` across `/v1/admin/...` and `/admin/v1/...` routes. Endpoint handlers should not hard-code their own required-justification checks; they may read the value for local logging, and should honor both `?justification=` and `X-Justification`.
**Capabilities auth rule**: The capabilities endpoints allow either a real resolved client identity or an authenticated admin/auditor role. Raw `X-Client-ID` is not a production credential; resolver support for that header is compiled only with `auth_testfixtures` and still requires `ModeTesting`.
**Raw X-Client-ID trust boundary**: Production route wrappers must rely on `AuthMiddleware` / `TokenResolver` for client identity. Caller-supplied `X-Client-ID` must be ignored in default builds and accepted only by binaries built with `auth_testfixtures` and running in `ModeTesting`.
**Mongo outbox staging rule**: Mongo now shares the same `AppendOutboxEvents(...)` write-path interface as the relational stores so handlers do not fork by datastore, but `MongoStore.InWriteTx` is still intent-only. Treat Mongo outbox appends as best-effort until the store is upgraded to `mongo.Session` transactions plus change-stream replay support.
**Postgres eventbus subscribe activation**: `internal/plugin/eventbus/postgres` must not return from `Subscribe(...)` until the refreshed LISTEN channel set for that subscription is active. Without the activation wait, tests and callers can race a stale broadcast-only LISTEN loop and miss recovery invalidation after forced subscription loss.
**SQLite BDD runner set**: SQLite coverage uses `internal/bdd/cucumber_sqlite_test.go`, `internal/bdd/cucumber_sqlite_encrypted_test.go`, and `internal/bdd/cucumber_sqlite_vec_test.go` plus `SQLiteTestDB`. There is no SQLite OIDC/Keycloak runner in this repo slice.
**Build capability gating**: `internal/buildcaps` exposes runtime booleans for optionally compiled modules (`SQLite`, `SQLiteFTS5`, `PostgreSQL`, `MongoDB`, `Redis`, `Infinispan`, `Qdrant`, `S3`, `AWSKMS`, `Vault`, `OpenAI`, `MCP`, `TCPListener`, `UDSListener`). Prefer using those in tests to skip capability-dependent suites/scenarios; keep build tags only where code would not compile without the optional module.
**SQLite FTS BDD policy**: SQLite BDD runners now compile whenever SQLite is present. Scenarios that specifically require SQLite FTS5 are tagged `@requires-sqlite-fts5` and the SQLite runners exclude them automatically when `buildcaps.SQLiteFTS5` is false.
**MCP BDD coverage style**: MCP BDD scenarios should launch the server the same way agents do, via `go run <projectDir> mcp ...` subprocesses speaking newline-delimited JSON-RPC over stdio, rather than calling tool handlers directly in-process.

**Search route parity behavior**: Go `POST /v1/conversations/search` now supports `searchType`, `afterCursor`, and `groupByConversation`; `searchType` accepts a string or array (e.g. `["semantic","fulltext"]`) and applies `limit` per requested type. Multi-type pagination uses an opaque base64url JSON cursor carrying per-type sub-cursors.
**Mongo conversation list query gap**: `MongoStore.ListConversations` currently ignores the `query` parameter for `GET /v1/conversations`; only the Postgres store applies title filtering there. `GET /v1/admin/conversations` does not support a free-text query in either store.
**Conversation list ordering**: `GET /v1/conversations` and `GET /v1/admin/conversations` now return newest-first (`created_at DESC`). Their `afterCursor` semantics follow that descending order, so the next page contains older conversations. Admin `mode=latest-fork` still collapses to one conversation per `conversation_group_id` after admin filters are applied. BDD coverage should assert exact counts and expected index order, not just `data[0]`.
**gRPC admin conversation payloads**: `AdminConversationsService` uses admin-specific proto messages for conversation details/summaries so `client_id`, `agent_id`, `metadata`, and response-in-progress state remain available to admin callers without exposing `client_id` on public conversation messages.
**Synthetic conversation archive field**: Conversation REST list/detail payloads expose synthetic `archived` instead of store-level `archivedAt`. Store summary/detail builders must preserve `ArchivedAt` all the way to the route serializer or archive filters can appear correct while the response still reports `archived: false`.
**OpenAPI conversation summary gotcha**: `contracts/openapi/openapi.yml` `ConversationSummary` itself must declare `archived`; putting the field only on the full `Conversation` schema makes wrapper-native `/v1/conversations` responses drop it when `internal/generated/api` is regenerated.
**Qdrant payload path gotcha**: Qdrant treats dots in filter keys as nested payload paths. Episodic memory vector payloads must store policy attributes under a nested `policy_attributes` object so filters such as `policy_attributes.topic` match; storing literal payload keys like `policy_attributes.topic` does not satisfy those filters.
