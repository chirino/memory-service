## What the Quarkus Extension Adds

The Quarkus integration now supports sub-agent orchestration in two places:

- `SubAgentTaskTool` is a reusable base class for app-specific tools such as `FactFindingSubAgentTool`.
- `StreamingSubAgentTaskTool` is the companion base class for app-specific streaming child tools such as `FeedbackSubAgentTool`.
- your app-specific tool binds a concrete child AI service while the extension owns joined execution, history appends, and task tracking.
- `SubAgentTurnRunner` keeps the parent turn open until joined child results have been added back into context and the LLM has had a chance to continue.
- `MemoryServiceProxy` exposes `GET /v1/conversations/{conversationId}/children`.

This keeps the user code narrow: the app supplies the parent AI service and the child AI service. The extension handles the joined task lifecycle.

## Try It With Curl

Define a helper to get a bearer token for `bob`:

```bash
function get-token() {
  curl -sSfX POST http://localhost:8081/realms/memory-service/protocol/openid-connect/token \
    -H "Content-Type: application/x-www-form-urlencoded" \
    -d "client_id=memory-service-client" \
    -d "client_secret=change-me" \
    -d "grant_type=password" \
    -d "username=bob" \
    -d "password=bob" \
    | jq -r '.access_token'
}
```

Create a parent conversation and ask the agent to split the work into child tasks. This prompt is written to strongly encourage the model to start one fact-finding child per CRM and then use the feedback child to evaluate the full set of findings:

```bash
curl -NsSfX POST http://localhost:9090/chat/5a7129d8-0a9f-4688-b650-c085d3f30c13 \
  -H "Content-Type: text/plain" \
  -H "Authorization: Bearer $(get-token)" \
  -d "Evaluate Salesforce, HubSpot, and Zoho CRM for a mid-sized B2B sales team. Start a fact-finding sub-agent for each CRM to gather pricing, security, API integration, and customization facts. Run those fact-finding tasks in parallel if possible. Then start a feedback sub-agent to review the findings across all three CRMs together, identify gaps or weak comparisons, and produce follow-up research questions for the fact-finding sub-agents before giving a final recommendation."
```

If the model decides to use `messageSubAgent`, the extension will create child conversations when needed, wait for those joined tasks before finalizing the parent turn, add the child results back into the parent context, and then let the parent LLM continue.

List the direct children that were created for the parent conversation:

```bash
curl -sSfX GET http://localhost:9090/v1/conversations/5a7129d8-0a9f-4688-b650-c085d3f30c13/children \
  -H "Authorization: Bearer $(get-token)" | jq
```

You should see one item per child conversation. Each child ID is the conversation ID returned by `messageSubAgent`.

Capture one of the child IDs so you can inspect its activity:

```bash
CHILD_ID=$(
  curl -sSfX GET http://localhost:9090/v1/conversations/5a7129d8-0a9f-4688-b650-c085d3f30c13/children \
    -H "Authorization: Bearer $(get-token)" \
  | jq -r '.data[0].id'
)
```

Inspect the child entries and their agent attribution:

```bash
curl -sSfX GET "http://localhost:9090/v1/conversations/${CHILD_ID}/entries" \
  -H "Authorization: Bearer $(get-token)" | jq
```

In the child history you should see the delegated user task followed by the sub-agent response. If your application sets `agentId` values, those show up here as well.

Inspect the parent conversation again to see how the joined results were folded back into the same user turn:

```bash
curl -sSfX GET http://localhost:9090/v1/conversations/5a7129d8-0a9f-4688-b650-c085d3f30c13/entries \
  -H "Authorization: Bearer $(get-token)" | jq
```

The parent conversation should now include the original user request, one or more internal join-result prompts added by the runtime, and the final AI response that was only returned after the joined child tasks had completed.
